 There is a critical multi-tenancy bug. When an admin uses
  "Member View" to impersonate a member, the Campaigns page, Link
   Tracking Settings page, and Lead Analytics all show the SAME
  data (the admin's data) regardless of which member is selected.
   Every member sees everyone else's campaigns, links, and
  settings.

  Root cause: The campaign/link-tracking API routes are not using
   resolveUserFromSession() from src/lib/session-utils.ts to get
  the effective user ID. They are either using the raw session
  (which returns the admin's ID) or not filtering by userId at
  all.

  The schema is correct — Campaign, TrackingLink, Click, and
  Conversion all have proper userId foreign keys. The bug is
  purely in the API routes.

  What to fix:

  1. Find every API route under src/app/api/ related to
  campaigns, tracking links, link tracking settings, clicks,
  conversions, and lead analytics. For each one:
    - Import resolveUserFromSession from @/lib/session-utils
    - Replace any auth() + session.user.id pattern with const
  user = await resolveUserFromSession()
    - Use user.id in all Prisma where clauses to filter data
  2. The correct pattern already exists in
  src/app/api/ai-tools/saved-scripts/route.ts — copy that
  approach:
  import { resolveUserFromSession } from "@/lib/session-utils";

  const user = await resolveUserFromSession();
  if (!user) return NextResponse.json({ error: "Unauthorized" },
  { status: 401 });

  // Then use user.id in all queries:
  const campaigns = await prisma.campaign.findMany({
    where: { userId: user.id },
    // ...
  });
  3. The tracking snippet data-id must also be per-member. Right
  now every member gets the same data-id value in their landing
  page snippet. Each member needs their own unique identifier so
  clicks are attributed to the correct member.
  4. The thank you page URL must be stored per-member (in the
  User model or a separate MemberSettings model) and fetched
  using resolveUserFromSession() so each member sees and saves
  their own URL.
  5. When creating campaigns or tracking links, always set
  userId: user.id using the resolved user from
  resolveUserFromSession().
  6. The stats cards (Clicks 30d, Leads 30d, Conv. Rate, Top
  Link) must also filter by userId — right now they show
  aggregate numbers across all members.

  resolveUserFromSession() already handles impersonation
  correctly — when an admin is in Member View, it reads the
  abv-impersonate-id cookie and returns that member's ID instead
  of the admin's. So using it everywhere automatically fixes both
   the member's own view AND the admin's Member View.

  Lead tracking bug found. Here's what's happening:                                         
                                                                                            
  The member's form submission carries the ?ref= parameter through to the thank you page    
  URL:                                                                                      
  https://www.chamberlaingroup.ca/thank-you/?ref=SIqCV3ik                                   
                                                                                            
  So when t.js loads on the thank you page, it sees ?ref= and treats it as a NEW click      
  instead of a lead conversion. This inflates click counts and never records leads.         
                                                                                            
  === FIX THE LOGIC IN t.js ===                                                             
  
  The snippet needs smarter logic:                                                          
                                                                                          
  1. When t.js loads and sees ?ref= in the URL:                                             
     - FIRST check if this page matches the thank you page path (server-side check via the
  pageview/click endpoint)                                                                  
     - If the server responds that this IS the thank you page, record it as a LEAD (not just
   a click)                                                                                 
     - The click should still be recorded too (the initial click is valid), but the lead  
  must also be created                                                                      
                                                                                          
  2. Actually, the simpler fix: the server-side thank you page matching on the              
  /api/tracking/click endpoint should handle this. When a click comes in and the page_url 
  matches the thank you page path, create BOTH a Click record AND a Lead record in the same 
  request.                                                                                

  === FIX PATH MATCHING ===

  The thank you page URL is: https://www.chamberlaingroup.ca/thank-you/?ref=SIqCV3ik        
  The saved path might be: /thank-you
                                                                                            
  The path matching must:                                                                   
  - Strip query parameters before comparing (ignore ?ref=xxx)                               
  - Handle trailing slashes (match /thank-you AND /thank-you/)                              
  - Compare pathname only (ignore domain)                                                   
  - Be case-insensitive                                                                     
                                                                                            
  So when comparing, extract just the pathname from the page_url, strip trailing slash,     
  lowercase it, and compare to the saved thankYouPageUrl (also stripped and lowercased).    
                                                                                            
  === FIX CROSS-DOMAIN SESSION ===                                                        

  The destination URL is on guides.chamberlaingroup.ca but the thank you page is on         
  www.chamberlaingroup.ca. The session cookie set on guides.chamberlaingroup.ca won't be
  readable on www.chamberlaingroup.ca because they're different subdomains.                 
                                                                                          
  Two options (pick the simpler one):                                                       
  Option A: Since the ?ref= param carries through to the thank you page anyway, the snippet
  can just use the ref code directly from the URL — no cookie needed to attribute the lead. 
  Option B: Set the cookie domain to .chamberlaingroup.ca (dot prefix) so it's shared across
   all subdomains.                                                                          
                                                                                          
  Option A is more reliable since the ref code is right there in the URL. The click endpoint
   already handles ?ref= — just make sure it also creates the lead when the page matches the
   thank you path.

  The fix is: when /api/tracking/click receives a request where the page_url pathname
  matches the member's thank you page path (after stripping query params and trailing

  where ?ref= carries through to the thank you page.
